Monitor security settings

The tenant-wide drift surface: settings across M365, Entra, Exchange, Teams and SharePoint — read, compared, and guarded.

CA and Intune policies are objects you deploy. But a tenant's security posture also lives in dozens of scattered switches: Exchange transport settings, Teams external access, SharePoint sharing, Entra defaults, anti-phishing policies. The Security page (sidebar → Policies → Security) is where Panoptica365 reads those switches across every tenant and tells you when one flips.

What you see

Pick a tenant and you get the monitored settings, grouped by category, with filter chips:

  • Category: All / Exchange / Identity / SharePoint / Teams / Compliance.
  • Priority: All / Critical / High / Medium / Low.

Each setting row shows its name, the current live value (interpreted into plain language, not raw API output), the license it requires if any, and a status. The status is actual compliance, re-evaluated on every poll — you don't configure anything to make a setting "count":

StatusWhat it means
Compliant — monitored (green)The live value already matches the recommended value — or a value you accepted as this tenant's baseline. Guarded from here on. No action needed; it turns green on the next poll on its own.
Off recommended — review (orange)A readable value that doesn't match the recommendation, and you haven't accepted it. A flag to review — it does not alert.
Not configured — nothing to monitor (grey)Nothing to evaluate: the underlying object doesn't exist or can't be read (a tenant with no DLP policies, an email-protection preset never turned on). No alert.
Drift Detected (red)A setting that was compliant or accepted has since changed. This fires a security-drift alert through the normal alert pipeline.
Poll Error (grey)The reader couldn't fetch the value — usually a license or permission gap.

Click a setting for the detail view: what the setting does, why it matters, the user impact of changing it, operator notes, and the expected vs actual values when drifted.

Applying and accepting

First, the change that matters most: the status reflects actual compliance, re-evaluated on every poll. You no longer set a desired state to make a setting "count" — a tenant that already matches the recommendation turns green on its own, with nothing to click. Two verbs remain, and you reach for them only on an orange (off-recommended) setting:

  • Apply — push the recommended value to the tenant. Use it on an orange setting you want to bring into line. Applies run asynchronously: the job is queued, a worker executes it, and the row updates when it completes (with a refresh check shortly after to confirm the value stuck). You can keep working while it runs.
  • Accept current as baseline — for an orange setting whose non-recommended value is intentional for this tenant, adopt the live value as the monitored baseline. The setting turns green and is guarded against that accepted value from then on; if it later changes, it goes red and alerts. This is how a legitimately non-standard tenant lives without a permanent orange flag.

The distinction matters during onboarding. A neglected tenant's orange settings get Apply — impose your standard. A well-configured tenant that legitimately runs a different value gets Accept — capture that reality as its baseline. Either way the setting ends up green and guarded, and any future deviation produces a drift alert.

Audit-only settings

A few settings are deliberately audit-only — Panoptica365 reads them but will not write them, typically because the write is license-gated or too sharp-edged to automate (DLP configuration is the canonical example). For these, you capture a baseline of the current configuration; from then on, any change to it alerts: "Baseline captured. Panoptica365 will alert on any DLP configuration change going forward." Remediation, when needed, is done by hand in the Microsoft console.

Recommended values

Setting detail views describe the recommended posture, but "most secure" is not "universally correct" — some settings legitimately vary with the customer's business model (external sharing for a company that collaborates with clients in SharePoint, for example). The recommendation text says for whom a value is recommended. Read it before bulk-applying anything.

Where this shows up elsewhere
